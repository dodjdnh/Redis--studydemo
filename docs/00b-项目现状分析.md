# 00b-项目现状分析

扫描目标项目：`C:\Users\Kevin\Desktop\IM\InfiniteChat\RealTimeCommunicationService`

扫描时间：2026-04-17

## 1. 扫描结论概览

当前 RealTimeCommunicationService 已经接入 Redis，但 Redis 主要用于“消息投递状态追踪”和“重试任务互斥锁”，还没有形成完整的 IM Redis 设计层。

项目依赖的是 Spring Data Redis，底层默认使用 Lettuce；没有发现 Redisson 依赖，也没有发现自定义 RedisConfig、Pub/Sub、Stream、布隆过滤器、两级缓存等实现。

在线用户状态目前保存在 JVM 内存里的 `ChannelManager`，不是 Redis。这个设计在单实例学习阶段可以跑通，但一旦服务多实例部署，不同节点之间无法知道“用户在哪台机器上在线”，消息推送会出现跨节点不可见的问题。

## 3. 现有 Redis 用法清单

| 功能点 | 使用的数据类型 | Key 设计 | 有无 TTL | 潜在问题 |
|--------|----------------|----------|----------|----------|
| 单条消息的投递状态集合 | Hash | `RTC:DELIVERY:{messageId}`，field 是 `userId`，value 是 `MessageDeliveryStatusVO` JSON | 有，7 天 | `getStatuses(messageId)` 使用 `HGETALL` 读取整条消息的所有收件人状态；群消息收件人很多时会有大 Hash 和阻塞风险 |
| 用户到消息的反向索引 | Set | `RTC:DELIVERY:USER:{userId}`，member 是 `messageId` | 有，7 天 | `getStatusesByUser(userId)` 使用 `SMEMBERS` 一次拉全量；用户 7 天内消息很多时会形成大 Set |
| 消息快照 | String | `RTC:MESSAGE:{messageId}`，value 是 `Message` JSON | 有，7 天 | 适合重试恢复，但只是快照，不是可靠消息队列；过期后重试会进入死信 |
| 可重试投递索引 | Set | `RTC:DELIVERY:RETRY`，member 是 `messageId|userId` | 有，7 天 | `getRetryableStatuses()` 使用 `SMEMBERS` 扫全量；重试量大时会拖慢定时任务，且没有按 `nextRetryAt` 排序 |
| 死信投递索引 | Set | `RTC:DELIVERY:DEAD`，member 是 `messageId|userId` | 有，7 天 | `getDeadStatuses()` 使用 `SMEMBERS` 扫全量，再在 JVM 内存中过滤/排序；数据多时分页不是真分页 |
| 死信总统计 | Hash | `RTC:DELIVERY:DEAD:STATS`，field 包括 `total`、`reason:{code}` | 无 | 总统计永久累积，没有保留策略；长期运行后字段数量取决于 reason 维度 |
| 死信日统计 | Hash | `RTC:DELIVERY:DEAD:STATS:{yyyy-MM-dd}` | 无 | 每天一个 key，但没有 TTL；如果运行多年会持续增长 |
| 重试任务分布式锁 | String | `RTC:DELIVERY:RETRY:LOCK`，value 是 UUID token | 有，默认 60000 ms | 手写 `SET NX PX` 风格锁，没有自动续期；重试任务超过租约时，其他实例可能同时进入重试 |

## 4. 分布式锁现状

当前锁实现位置：

- `MessageDeliveryRetryServiceImpl#acquireRetryLock`
- `MessageDeliveryRetryServiceImpl#releaseRetryLock`

当前实现方式：

- 使用 `StringRedisTemplate.opsForValue().setIfAbsent(key, token, leaseMillis, TimeUnit.MILLISECONDS)` 抢锁。
- 使用 Lua 脚本比较 token 后再 `DEL`，避免误删别的实例持有的锁。

这个释放方式是正确方向，因为 `DefaultRedisScript` 可以把“判断 token + 删除 key”放在 Redis 内原子执行，避免先 GET 再 DEL 的竞态。

主要问题：

- 没有 Redisson，看不到 `RLock` 看门狗自动续期。
- `lockLeaseMillis` 默认 60000ms，如果一次重试扫描和推送超过 60 秒，锁会提前过期，第二个实例可能进来重复重试。
- 锁粒度是全局重试锁 `RTC:DELIVERY:RETRY:LOCK`，实现简单，但会让所有实例串行重试，后续数据量大时吞吐受限。

后续建议：

- 阶段二先讲清楚当前 SETNX 锁的边界，再给出 Redisson `RLock` 版本。
- 如果继续保留手写锁，至少要把单次扫描量限制住，避免一次任务超过 lease 时间。

## 5. 缓存策略现状

当前没有发现典型业务缓存层：

- 没有 `@Cacheable`、`@CacheEvict`。
- 没有用户信息缓存、群组信息缓存、会话信息缓存。
- 没有缓存穿透防护，例如布隆过滤器或空值缓存。
- 没有缓存击穿防护，例如热点 key 互斥锁。
- 没有缓存雪崩防护，例如随机 TTL 抖动。
- 没有 Caffeine + Redis 两级缓存。

这说明项目现在使用 Redis 更像“状态存储/索引存储”，还没有把 Redis 用作完整缓存体系。

后续阶段三应该重点围绕 IM 项目的用户信息、群组信息、会话校验结果来补缓存设计，而不是泛泛讲三大缓存问题。

## 6. 消息相关 Redis 用法

### 6.1 当前已有能力

当前消息投递链路已经用 Redis 做了这些事：

- `recordPending(message)`：记录每个收件人的初始投递状态。
- `markSent(...)`：Netty 写成功后标记为 `SENT`。
- `markAcked(...)`：客户端 ACK 后标记为 `ACKED`。
- `markOffline(...)` / `markFailed(...)`：用户离线或发送失败时进入可重试索引。
- `markDead(...)`：超过重试次数或快照缺失时进入死信索引，并累加死信统计。

这部分是项目里 Redis 用得最接近 IM 业务核心的地方。

### 6.2 当前缺失能力

- 没有 Redis Pub/Sub：多实例之间不能广播“某个房间/用户有新消息需要本机推送”。
- 没有 Redis Stream：离线消息和可靠投递没有消费者组、ACK、Pending List 这些机制。
- 离线消息目前不是独立队列，而是通过投递状态和消息快照间接支持重试。
- 在线状态只看本机 `ChannelManager`，跨实例时 `isChannelActive(targetUserId)` 只能判断本机连接。

### 6.3 对 IM 场景的影响

现在单实例下的消息推送逻辑是清晰的；但如果有两个 RTC 实例：

```text
用户 A 连接 RTC-1
用户 B 连接 RTC-2
用户 A 给 B 发消息，请求打到 RTC-1
RTC-1 的 ChannelManager 查不到 B
于是 B 被误判为 offline
```

这就是后续阶段四、阶段五必须处理的核心问题：在线推送不能只依赖本机内存，需要 Redis 保存在线状态或服务节点路由，再配合 Pub/Sub 做跨节点通知。

## 7. 在线状态现状

在线用户当前由 `ChannelManager` 管理：

```java
private static final ConcurrentMap<String, Channel> USER_CHANNEL_MAP = new ConcurrentHashMap<>();
private static final ConcurrentMap<Channel, String> CHANNEL_USER_MAP = new ConcurrentHashMap<>();
```

连接建立时：

- `MessageInboundHandler#handleHandshakeComplete` 把 `userId -> Channel` 放入内存 Map。

连接断开时：

- `MessageInboundHandler#offline` 从内存 Map 移除。

心跳处理：

- `MessageInboundHandler#processHeartBeat` 只回一个心跳响应，没有续期 Redis 在线状态。

问题：

- 进程重启后在线状态全部丢失。
- 多实例之间在线状态不可见。
- 没有 Redis TTL 心跳模型，无法用过期时间自然兜底异常断线。

后续阶段五建议把在线状态设计为：

- `user:online:{userId}`：String，value 存服务实例 ID 或 channel 标识，TTL 由心跳续期。
- `rtc:node:users:{nodeId}`：Set，记录本节点连接用户，便于节点下线清理。

## 8. 当前项目最薄弱的 3 个 Redis 点

### 8.1 在线状态没有进 Redis

这是最影响 IM 架构扩展的点。当前 `ChannelManager` 是单 JVM 内存结构，只适合单实例。后续要把“用户是否在线”和“在哪个服务节点在线”放进 Redis，否则多实例部署时会误判离线。

### 8.2 大 Set / 大 Hash 查询风险

当前这些方法都有全量读取风险：

- `opsForHash().entries(buildKey(messageId))`
- `opsForSet().members(buildUserIndexKey(userId))`
- `opsForSet().members(RETRY_INDEX_KEY)`
- `opsForSet().members(DEAD_INDEX_KEY)`

这些方法在数据小时很好写，但在 IM 场景中，消息数、用户数、失败记录都可能快速增长。阶段一需要重点讲 `HSCAN` / `SSCAN`、分页索引和 ZSet 时间排序，避免后续做成大 Key。

### 8.3 重试索引用 Set，不适合按时间调度

当前 `RTC:DELIVERY:RETRY` 是 Set，重试时需要全量扫出来，再逐条判断 `nextRetryAt`。

更适合的结构是 ZSet：

- key：`RTC:DELIVERY:RETRY`
- member：`messageId|userId`
- score：下一次可重试时间戳

这样可以用 `ZRANGEBYSCORE` 只取“现在应该重试”的记录，避免每 30 秒全量扫描。

注意：ZSet 的 score 是 double，不能把超大整数或需要严格唯一顺序的消息序号直接当 score；但作为毫秒级调度时间戳可以用，真正消息序号应该用 `INCR`。

## 9. 后续学习重点建议

结合当前代码，后续五个阶段建议这样对齐项目：

| 阶段 | 本项目对应重点 |
|------|----------------|
| 阶段一：数据类型精准选型 | 重点改造 `RTC:DELIVERY:RETRY`、`RTC:DELIVERY:USER:{userId}`、在线状态 Key 设计，讲清楚 Set/Hash/ZSet/String 在这些场景下怎么选 |
| 阶段二：分布式锁进阶 | 对比当前 `RTC:DELIVERY:RETRY:LOCK` 手写锁和 Redisson `RLock`，重点解释业务执行超过锁 TTL 的并发问题 |
| 阶段三：缓存设计与陷阱 | 给用户信息、会话校验、群组资料补缓存穿透/击穿/雪崩防护，而不是停留在概念 |
| 阶段四：Pub/Sub 与 Stream | 设计“在线推 Pub/Sub，离线/可靠投递用 Stream 或持久化队列”的消息链路，解决多实例推送不可见问题 |
| 阶段五：IM 实战落地 | 把在线状态 TTL、消息序号 `INCR`、未读数 `HINCRBY`、跨节点房间广播整合进 RTC 服务 |

## 10. 下一步

项目现状已经明确：当前 Redis 不是空白，已经承担了消息投递状态和重试控制，但还缺少生产级 IM 必需的在线状态、跨节点推送、可靠队列、大 Key 防护和缓存防护。

下一步进入第 0 步知识摸底，重点不问基础数据类型，而是围绕当前项目最可能踩坑的四个问题：ZSet score、锁超时、缓存三大问题、Pub/Sub 和 Stream 的取舍。
