## 1. 为什么 IM 项目特别依赖 Redis

IM 项目不是“用了 Redis 会更快”这么简单，而是很多能力如果只靠 MySQL 或 JVM 内存，架构上会出问题。

### 1.1 高并发读写

在线状态、未读数、消息投递状态这类数据更新非常频繁。

比如一个用户每 30 秒发一次心跳，如果有 10 万在线用户，就会持续产生大量更新。MySQL 适合保存长期可靠数据，但不适合承担这种高频状态刷新。

Redis 更适合保存这类“变化快、访问频繁、可以过期”的状态。

### 1.2 实时性要求

消息推送、在线判断、未读数变化都需要毫秒级响应。

如果每次判断用户是否在线都查数据库，消息链路会变慢。Redis 可以把这些实时状态放在内存里，读写更快。

### 1.3 分布式协调

当前 RTC 项目里的在线用户状态存在 `ChannelManager` 这个 JVM 内存 Map 里。

单实例时没问题，但多实例时会出现：

```text
用户 A 连接 RTC-1
用户 B 连接 RTC-2
用户 A 给 B 发消息，请求打到 RTC-1
RTC-1 的内存里没有 B
RTC-1 误以为 B 离线
```

Redis 可以作为多台 RTC 服务共享的状态中心，保存“用户是否在线”和“用户在哪个节点在线”。

## 2. Redis 在 IM 里的第一层理解

先不背数据类型定义，先从 IM 场景理解 Redis 存什么。

| IM 场景 | Redis 里存什么 | 为什么适合放 Redis |
|--------|----------------|--------------------|
| 用户在线状态 | `userId -> 在线节点 / channel 信息` | 状态变化快，断线后可以用 TTL 自动过期 |
| 用户 Session 信息 | `token -> 用户身份、设备、过期时间` | 登录态访问频繁，需要快速校验 |
| 离线消息 | `userId -> 待补推消息列表` | 用户上线后快速拉取 |
| 房间成员 | `roomId -> 成员集合` | 需要快速判断某人是否在房间里 |
| 最近联系人 | `userId -> 联系人 + 最近活跃时间` | 需要按时间排序 |
| 未读数 | `userId + roomId -> 未读数量` | 需要原子加减 |
| 消息投递状态 | `messageId -> 每个收件人的状态` | 当前 RTC 项目已经在做 |
| 死信统计 | `reason -> 失败次数` | 当前 RTC 项目已经在做 |

TTL 是 Redis 的过期时间机制：给 key 设置一个存活时间，到期后 Redis 自动删除它。IM 里常用它做在线状态兜底，比如用户异常断网后，心跳不再续期，在线 key 自动消失。

## 3. IM 场景下的数据类型选型速查表

| 场景 | 推荐类型 | Key 设计示例 | 选择原因 |
|------|----------|--------------|----------|
| 用户在线状态 | String | `user:online:{userId}` | 只需要存一个值，比如节点 ID，配合 TTL 就够 |
| 用户 Session 信息 | Hash | `session:{token}` | 多字段对象，适合单独更新某个字段 |
| 离线消息队列 | List | `offline:msg:{userId}` | List 适合按顺序放入、按顺序取出 |
| 房间成员列表 | Set | `room:members:{roomId}` | Set 天然去重，适合判断用户是否在房间 |
| 最近联系人 | ZSet | `recent:contacts:{userId}` | ZSet 可以按 score 排序，score 通常放时间 |
| 未读消息计数 | Hash | `unread:{userId}` | field 放 roomId，value 放未读数，适合按房间加减 |
| 全局消息序号 | String | `msg:seq:{roomId}` | 用 `INCR` 原子自增生成不会重复的序号 |
| 消息投递状态 | Hash | `RTC:DELIVERY:{messageId}` | 当前项目就是每个 userId 一个投递状态 |
| 重试调度索引 | ZSet | `RTC:DELIVERY:RETRY` | score 放下次重试时间，只取到期任务 |

`INCR` 是 Redis 的原子自增命令，多个线程同时执行也不会生成重复数字，适合做消息序号。

当前已经有的 Redis 使用：

| 功能 | 当前做法 | 后续学习关注点 |
|------|----------|----------------|
| 消息投递状态 | Hash：`RTC:DELIVERY:{messageId}` | 大 Hash 时不能随便 `HGETALL` |
| 用户消息索引 | Set：`RTC:DELIVERY:USER:{userId}` | 大 Set 时不能随便 `SMEMBERS` |
| 消息快照 | String：`RTC:MESSAGE:{messageId}` | 这是重试快照，不是可靠消息队列 |
| 重试索引 | Set：`RTC:DELIVERY:RETRY` | 更适合改成 ZSet 按时间调度 |
| 死信统计 | Hash：`RTC:DELIVERY:DEAD:STATS` | 需要考虑统计 key 的保留策略 |
| 重试锁 | String + `setIfAbsent` | 没有看门狗，任务超时会有并发风险 |
| 在线状态 | JVM 内存 `ChannelManager` | 多实例不可见，应该落 Redis |

`HGETALL` 是一次取出 Hash 里所有字段的命令，数据少时方便，数据大时会阻塞 Redis。

`SMEMBERS` 是一次取出 Set 里所有成员的命令，Set 很大时也会有阻塞风险。

## 5. 核心风险地图

```text
消息发送
    ↓
[风险1] 消息重复投递
    需要幂等性保证，例如消息 ID 去重或分布式锁
    ↓
[风险2] 离线消息丢失
    需要可靠存储，List 或 Stream 都可能用到
    ↓
[风险3] 用户状态不一致
    当前只在 JVM 内存里，后续要改成 Redis TTL 心跳
    ↓
[风险4] 热点缓存击穿
    热点用户或群组资料过期时，大量请求可能同时打到数据库
    ↓
[风险5] 消息序号乱序
    严格序号应该用 INCR，不要随便用 ZSet score 代替
```

## 6. 本次学习涉及的核心 API / 类 / 命令

| 类 / 注解 / 命令 | 是干什么的 |
|------------------|------------|
| `StringRedisTemplate` | Spring 提供的 Redis 操作类，专门读写字符串 key 和字符串 value |
| `RedisTemplate` | Spring 提供的通用 Redis 操作类，可以操作对象，但要配置序列化 |
| `EXPIRE` | 给 key 设置过期时间 |
| `TTL` | 查看 key 还剩多久过期 |
| `INCR` | 对数字字符串做原子自增，常用于生成消息序号 |
| `HINCRBY` | 对 Hash 某个字段做原子加减，适合未读数 |
| `HGETALL` | 一次读取 Hash 所有字段，大数据量时要谨慎 |
| `HSCAN` | 分批读取 Hash，避免一次性拉太多 |
| `SMEMBERS` | 一次读取 Set 所有成员，大数据量时要谨慎 |
| `SSCAN` | 分批读取 Set，避免大 Set 阻塞 |
| `ZRANGEBYSCORE` | 从 ZSet 里按 score 范围取数据，适合取到期重试任务 |
| `setIfAbsent` | Spring 对 Redis `SET NX` 的封装，常用于抢锁 |
| `DefaultRedisScript` | Spring 执行 Lua 脚本的类，用来保证多个 Redis 操作原子执行 |
| `RLock` | Redisson 的分布式锁接口，支持看门狗自动续期 |
| `RBloomFilter` | Redisson 的布隆过滤器，用于拦截明显不存在的数据请求 |
| `@Cacheable` | Spring 缓存注解，命中缓存时方法体不会执行 |
| `@CacheEvict` | Spring 删除缓存的注解，通常在更新数据后使用 |
| `StreamOperations` | Spring 操作 Redis Stream 的接口 |

## 8. redisdemo 项目的依赖参考

Spring Boot 项目里优先确认或添加这些依赖。

Spring Data Redis 用于基本 Redis 操作：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>
```

Redisson 用于分布式锁和布隆过滤器：

```xml
<dependency>
    <groupId>org.redisson</groupId>
    <artifactId>redisson-spring-boot-starter</artifactId>
    <version>3.23.5</version>
</dependency>
```

Caffeine 用于本地缓存，后续做两级缓存会用：

```xml
<dependency>
    <groupId>com.github.ben-manes.caffeine</groupId>
    <artifactId>caffeine</artifactId>
</dependency>
```

最小 Redis 配置：

```yaml
spring:
  redis:
    host: localhost
    port: 6379
    lettuce:
      pool:
        max-active: 8
        max-idle: 8
```

`lettuce.pool.max-active` 是 Lettuce Redis 客户端连接池的最大活跃连接数，高并发下默认值太小会导致请求排队。
