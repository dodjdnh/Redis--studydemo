## 2. 五个阶段完整链路

### 2.1 阶段一：数据类型精准选型

阶段一解决的是：

```text
IM 的不同业务数据，应该用 Redis 的哪种数据结构。
```

你掌握了：

| IM 场景 | Redis 结构 | 原因 |
|---------|------------|------|
| 在线状态 | String + TTL | 一个用户一个在线节点，心跳续期，超时自动离线 |
| Session 信息 | Hash + TTL | 一个 token 下有多个字段，适合局部更新 |
| 离线消息入门版 | List | 简单 FIFO 队列，但没有 ACK |
| 房间成员 | Set | 去重，快速判断成员是否存在 |
| 最近联系人 | ZSet | 按最近消息时间排序 |
| 消息序号 | String + `INCR` | 严格递增、不重复 |
| 未读数 | Hash + `HINCRBY` | 一个用户多个会话，每个会话一个计数 |

阶段一最重要的结论：

```text
不要背数据类型定义，要能从 IM 场景反推数据结构。
```

### 2.2 阶段二：分布式锁和幂等

阶段二解决的是：

```text
多实例并发下，如何避免重复执行和重复处理。
```

你掌握了：

| 技术 | 解决什么 |
------|----------|
| `setIfAbsent` | 用 Redis `SET NX` 抢锁 |
| token + Lua | 安全释放锁，避免误删别人的锁 |
| Redisson `RLock` | 更完整的分布式锁 |
| 看门狗 | 业务未结束时自动续期 |
| `tryLock` | 限时抢锁，抢不到就跳过 |
| `msg:dedup:{messageId}` | 同一消息只处理一次 |

阶段二对应 RTC 项目里的：

```text
RTC:DELIVERY:RETRY:LOCK
```

当前项目已有手写锁，但固定 TTL 有风险。后续可以升级为：

```text
Redisson tryLock + 看门狗
```

阶段二最重要的结论：

```text
分布式锁防止多个实例同时做同一件事。
消息幂等防止同一条消息被重复处理。
```

### 2.3 阶段三：缓存设计与陷阱

阶段三解决的是：

```text
Redis 作为缓存时，如何保护数据库和下游服务。
```

你掌握了三大缓存问题：

| 问题 | 一句话区分 | 解决方式 |
|------|------------|----------|
| 缓存穿透 | 查不存在的数据 | 布隆过滤器 + 空值缓存 |
| 缓存击穿 | 热点 key 过期 | 互斥锁 + 双重检查 |
| 缓存雪崩 | 大量 key 同时过期 | 随机 TTL + 分批预热 |

你还掌握了：

```text
Caffeine 本地缓存 + Redis 远程缓存
```

两级缓存适合：

```text
用户昵称
头像
群基础信息
系统配置
```

权限类数据要谨慎：

```text
是否在群
是否拉黑
是否禁言
```

阶段三最重要的结论：

```text
缓存不是简单 GET/SET。
缓存必须考虑不存在、热点过期、批量过期和多实例一致性。
```

### 2.4 阶段四：Pub/Sub 与 Stream

阶段四解决的是：

```text
在线消息如何跨节点通知，离线消息如何可靠保存。
```

你掌握了：

| 对比点 | Pub/Sub | Stream |
|--------|---------|--------|
| 是否持久化 | 不持久化 | 持久化 |
| 离线时 | 消息丢失 | 消息保留 |
| ACK | 不支持 | 支持 |
| 消费者组 | 不支持 | 支持 |
| 适合场景 | 在线通知 | 可靠投递 |

阶段四最重要的结论：

```text
Pub/Sub 负责在线实时通知。
Stream 负责离线可靠补推。
```

它们不是二选一，而是组合：

```text
在线用户：Pub/Sub 通知目标节点推 WebSocket
离线用户：Stream 保存消息，用户上线后读取并 ACK
```

### 2.5 阶段五：IM 实战落地

阶段五解决的是：

```text
把前四阶段组合成一套 IM Redis 设计。
```

你完成了：

| 能力 | Redis 设计 |
|------|------------|
| 用户在线状态 | `user:online:{userId}` String + TTL |
| 心跳续期 | `EXPIRE user:online:{userId}` |
| 消息序号 | `INCR msg:seq:{roomId}` |
| 未读数 | `unread:{userId}` Hash |
| 房间成员 | `room:members:{roomId}` Set |
| 房间广播 | `im:room:broadcast` Pub/Sub |
| 离线补推 | `offline:stream:{userId}` Stream |
| 消息幂等 | `msg:dedup:{messageId}` |

阶段五最重要的结论：

```text
Redis 在 IM 里不是单点工具，而是一组组合能力。
```

## 4. 与 RealTimeCommunicationService 的落地建议

### 4.1 在线状态

现状：

```text
ChannelManager 只存在 JVM 内存。
```

建议：

```text
WebSocket 握手成功 -> SET user:online:{userId} {nodeId} EX 90
心跳 -> EXPIRE user:online:{userId} 90
断开 -> DEL user:online:{userId}
```

`ChannelManager` 仍然保留，但只负责本机连接。

Redis 负责跨实例可见的在线状态。

### 4.2 消息发送

建议链路：

```text
1. 校验 messageId 幂等
2. 校验会话成员关系
3. INCR 生成 room seq
4. 保存消息
5. 记录投递状态
6. 查询接收方在线状态
7. 在线走 Pub/Sub
8. 离线写 Stream
9. 客户端 ACK 后更新状态
```

### 4.3 重试任务

现状：

```text
setIfAbsent + TTL + Lua
```

建议：

```text
Redisson tryLock(2, TimeUnit.SECONDS)
```

不要轻易指定 leaseTime，让看门狗处理续期。

### 4.4 重试索引

现状：

```text
RTC:DELIVERY:RETRY 使用 Set
```

风险：

```text
每次要全量扫描，再判断 nextRetryAt。
```

建议：

```text
ZSet
key = RTC:DELIVERY:RETRY
member = messageId|userId
score = nextRetryAt 时间戳
```

这样可以只取到期任务。

### 4.5 缓存

可以考虑缓存：

```text
user:info:{userId}
group:info:{groupId}
session:members:{sessionId}
```

但权限类缓存要注意删除和过期。

## 5. 生产环境要注意的事

### 5.1 大 Key 监控

单个 key 的 value 太大，会造成 Redis 阻塞和网络传输压力。

重点关注：

```text
room:members:{roomId}
unread:{userId}
RTC:DELIVERY:{messageId}
RTC:DELIVERY:USER:{userId}
```

可以用：

```text
redis-cli --bigkeys
```

扫描大 key。

### 5.2 热 Key 问题

如果某个 key 访问频率极高：

```text
user:info:10001
group:info:hot-room
session:members:hot-session
```

单 Redis 节点可能成为瓶颈。

解决方向：

```text
Caffeine 本地缓存
key 拆分
读写分离
Redis Cluster
```

### 5.3 慢查询日志

Redis 是单线程执行命令模型，慢命令会影响其他请求。

重点警惕：

```text
KEYS
SMEMBERS 大 Set
HGETALL 大 Hash
LRANGE 大 List
XRANGE 大 Stream
```

可以查：

```text
SLOWLOG GET 10
```

### 5.4 连接池配置

Lettuce 默认连接池配置较小，高并发下可能排队。

需要关注：

```yaml
spring:
  data:
    redis:
      lettuce:
        pool:
          max-active: 8
          max-idle: 8
          min-idle: 0
          max-wait: 100ms
```

生产环境要按并发量调整。

### 5.5 序列化一致性

`StringRedisTemplate` 简单直接，适合字符串。

如果使用 `RedisTemplate<Object, Object>`，必须明确序列化方式。

线上不要随便改序列化方式，否则旧数据可能无法反序列化。

### 5.6 Stream 裁剪策略

Stream 不能无限增长。

但 `XTRIM` 也不能乱用。

需要考虑：

```text
消息保留时间
最大长度
pending 消息
消费者组数量
失败补偿能力
```

### 5.7 Redisson 看门狗边界

看门狗只在不指定 leaseTime 时自动续期。

这些有看门狗：

```java
lock.lock();
lock.tryLock(2, TimeUnit.SECONDS);
```

这些没有看门狗：

```java
lock.lock(10, TimeUnit.SECONDS);
lock.tryLock(2, 10, TimeUnit.SECONDS);
```

### 5.8 密码不要写死

配置文件中不要直接写 Redis 密码。

建议：

```yaml
spring:
  data:
    redis:
      password: ${REDIS_PASSWORD:}
```

### 6.1 Redis Cluster


### 6.2 Redis 持久化

IM 场景要区分：

```text
哪些数据丢了可以接受
哪些数据丢了不能接受
```

需要理解：

```text
RDB
AOF
appendfsync
恢复时间
数据丢失窗口
```

### 6.3 Lua 脚本

Lua 可以把多个 Redis 命令合成一个原子操作。

适合：

```text
扣减不低于 0
校验并更新
批量状态切换
复杂幂等状态机
```

### 6.4 Redis + Kafka

生产级 IM 往往不会只靠 Redis 承担全部消息可靠性。

常见组合：

```text
Redis：在线状态、未读数、缓存、实时协调
Kafka：消息持久化、削峰、顺序消费、事件流
MySQL/ES：长期消息存储和查询
```

