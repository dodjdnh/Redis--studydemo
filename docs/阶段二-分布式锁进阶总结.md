# 阶段二-分布式锁进阶总结

## 1. 本阶段目标

阶段二解决的是：在多个服务实例同时运行时，如何避免同一批任务或同一条消息被重复处理。

这一阶段不是单纯学“怎么加锁”，而是围绕 RTC 项目的真实场景：

```text
RTC:DELIVERY:RETRY:LOCK
```

也就是失败消息重试任务，理解分布式锁为什么需要、手写锁有什么边界、Redisson 看门狗解决了什么，以及单条消息为什么还需要幂等保护。

## 4. IM 场景对应关系

### 4.1 失败消息重试任务

RTC 项目现在有定时任务扫描失败/离线消息并重试。

如果多个 RTC 实例都开启定时任务，不加锁会导致：

```text
RTC-1 扫到 msg-1 并重试
RTC-2 也扫到 msg-1 并重试
用户收到重复消息
```

所以需要分布式锁保证：

```text
同一时间只有一个实例执行重试扫描。
```

### 4.2 消息幂等

分布式锁锁住的是“一段任务”，但它不能完全替代消息幂等。

客户端重试、网络超时、服务端重复消费都可能让同一个 `messageId` 被处理多次。

所以单条消息还需要：

```text
msg:dedup:{messageId}
```

第一次处理写入成功，后续重复请求直接跳过。

## 5. 与项目现状的对比

| 项目现状 | 优点 | 风险 | 阶段二后的改进方向 |
|----------|------|------|--------------------|
| 当前使用 `setIfAbsent` 抢 `RTC:DELIVERY:RETRY:LOCK` | 能避免多个实例同时执行重试 | 锁 TTL 固定，业务执行超过 TTL 时会提前过期 | 用 Redisson `RLock` + 看门狗自动续期 |
| 当前释放锁使用 token + Lua | 避免误删其他实例的新锁 | 仍然无法解决业务超时导致的并发重入 | 保留 token 思路，或迁移到 Redisson |
| 当前抢不到锁直接跳过 | 定时任务不会阻塞 | 如果锁 TTL 太长，其他实例长时间无法执行 | 用 `tryLock(waitTime)` 最多等待短时间 |
| 当前消息投递状态能标记 ACK/SENT/OFFLINE | 能追踪投递结果 | 不能天然阻止同一 messageId 重复进入处理链路 | 增加 `msg:dedup:{messageId}` 幂等保护 |

## 6. 常见坑

### 6.1 释放锁不能直接 `DEL`

如果锁过期后被其他实例重新抢到，旧实例执行结束时直接 `DEL`，会删掉新实例的锁。

正确做法是：

```text
先比较 token，确认锁还是自己的，再删除。
```

这必须用 Lua 脚本保证原子性。

### 6.2 手写锁的 TTL 很难选

TTL 太短：

```text
业务没执行完，锁先过期，其他实例进来并发执行。
```

TTL 太长：

```text
实例抢锁后崩溃，其他实例要等很久才能继续执行。
```

这就是 Redisson 看门狗要解决的问题。

### 6.3 Redisson 指定 leaseTime 会关闭看门狗

这点非常容易踩坑。

有看门狗：

```java
lock.lock();
lock.tryLock(2, TimeUnit.SECONDS);
```

没有看门狗：

```java
lock.lock(10, TimeUnit.SECONDS);
lock.tryLock(2, 10, TimeUnit.SECONDS);
```

如果业务耗时不确定，不要随便指定 leaseTime。

### 6.4 `lock.lock()` 抢不到会一直等

定时任务不适合一直阻塞等待锁。

RTC 重试任务更适合：

```java
lock.tryLock(2, TimeUnit.SECONDS);
```

抢不到就跳过本轮，等下一轮调度。

### 6.5 浏览器刷新不一定制造并发

验证 `tryLock` 时，如果只在一个浏览器标签里连续刷新，请求可能被浏览器排队。

要看到“抢不到锁”，需要真正并发，比如两个浏览器、多个 PowerShell Job，或压测工具。

### 6.6 幂等 key 写成功但业务失败，会导致后续被跳过

当前练习版是：

```text
先写 msg:dedup:{messageId}
再处理业务
```

如果业务处理到一半失败，后续相同 messageId 会被当成重复消息跳过。

生产环境更稳的做法是：

```text
幂等 key value 存处理中 / 成功 / 失败状态
或者数据库对 messageId 建唯一索引兜底
```

## 7. 你现在应该掌握什么

你现在应该能解释：

```text
分布式锁：防止多个实例同时做同一件事。
消息幂等：防止同一条消息被重复处理。
```

也应该能区分：

| 技术 | 作用范围 | 生命周期 |
|------|----------|----------|
| 重试任务锁 | 一整个定时重试任务 | 很短，执行完释放 |
| 消息幂等 key | 单条 messageId | 较长，覆盖客户端重试窗口 |

对于 RTC 项目，阶段二的关键结论是：

```text
RTC:DELIVERY:RETRY:LOCK 可以从手写 setIfAbsent 锁升级为 Redisson tryLock + 看门狗。
消息发送入口可以增加 msg:dedup:{messageId}，防止重复消息重复处理。
```
