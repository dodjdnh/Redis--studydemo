## 3. IM 场景对应关系

| IM 场景 | 推荐 Redis 类型 | 本阶段代码 | 解决的问题 |
|---------|----------------|------------|------------|
| 用户 Session 信息 | Hash + TTL | `UserSessionStore` | 一个 token 下保存 userId、设备、节点、channel、活跃时间等多字段信息 |
| 离线消息队列 | List + TTL + 长度裁剪 | `OfflineMessageQueue` | 用户离线时先保存消息，上线后按顺序取出 |
| 房间成员管理 | Set | `RoomMemberManager` | 成员去重、快速判断用户是否属于房间 |
| 最近联系人排序 | ZSet | `RecentContactsRanking` | 会话按最近活跃时间排序 |
| 房间消息序号 | String + `INCR` | `RecentContactsRanking#nextMessageSeq` | 生成严格递增、不重复的消息序号 |
| 未读消息数 | Hash + `HINCRBY` | `UnreadCountService` | 每个用户按会话保存未读数，并支持原子加一 |

## 4. 与 RTC 项目现状的对比

| 现状 | 问题 | 阶段一后的理解 |
|------|------|----------------|
| 在线用户存在 `ChannelManager` 的 JVM 内存里 | 多实例之间不可见，跨节点推送会误判离线 | 在线状态应该逐步改成 Redis String + TTL，心跳续期 |
| `RTC:DELIVERY:{messageId}` 用 Hash 保存投递状态 | `HGETALL` 在收件人很多时有大 Hash 风险 | Hash 适合多字段状态，但大 Hash 要考虑 `HSCAN` |
| `RTC:DELIVERY:USER:{userId}` 用 Set 保存用户消息索引 | `SMEMBERS` 会一次拉全量，用户消息多时风险大 | Set 适合去重索引，但大集合遍历要用 `SSCAN` |
| `RTC:DELIVERY:RETRY` 用 Set 保存重试任务 | 每次重试需要全量扫描，再判断 `nextRetryAt` | 重试调度更适合 ZSet，score 存下次重试时间 |
| 还没有未读数 Redis 设计 | ACK 和投递状态不等于未读数 | 未读数适合 `unread:{userId}` Hash + `HINCRBY` |
| 还没有房间成员 Redis 缓存 | 每次权限校验都依赖远程服务 | 房间成员可用 Set 缓存，但成员变更必须同步删除/刷新 |

## 5. 常见坑

### 5.1 不要把 Redis 当成“随便存 JSON 的地方”

JSON String 适合整体读写，但 Session、未读数这种经常局部更新的对象，更适合 Hash。

例如只更新 `lastActiveAt` 时，Hash 只需要改一个 field；JSON String 则要整体读出、反序列化、修改、再整体写回。

### 5.2 大 Set 不要直接 `SMEMBERS`

房间成员、用户消息索引这类 Set 可能越来越大。

小集合可以 `SMEMBERS`，但大集合应该用 `SISMEMBER` 做单点判断，用 `SSCAN` 做分批遍历。

### 5.3 List 简单，但不是可靠消息队列

`RPOP` 是取出即删除。

如果服务刚取出离线消息，还没推送给 WebSocket 就崩溃，这条消息会丢。生产级可靠投递后面要看 Stream 的 ACK 机制。

### 5.4 ZSet score 适合排序，不适合严格消息序号

最近联系人可以用时间戳当 score 排序。

但消息序号要求严格递增、不重复，应该用 `INCR`，不要用 ZSet score 硬凑。

### 5.5 未读数不要先 `HGET` 再 `HSET`

高并发下，两个线程可能都读到旧值，然后互相覆盖。

未读数加一应该直接用 `HINCRBY`，让 Redis 保证原子性。

### 5.6 TTL 是设置在整个 key 上

Hash 里的单个 field 没有独立 TTL。

例如 `session:{token}` 过期时，里面的 `userId`、`device`、`lastActiveAt` 会一起消失。

## 6. 本阶段你现在应该掌握什么

你现在不需要死记 Redis 数据类型定义，但应该能说出这些选择：

```text
在线状态：String + TTL
Session：Hash + TTL
离线消息：List，但可靠性不够
房间成员：Set
最近联系人：ZSet
消息序号：String + INCR
未读数：Hash + HINCRBY
```

更重要的是，你应该能解释：

- 为什么在线状态不能只放 JVM 内存。
- 为什么大集合不能全量读取。
- 为什么未读数要用原子自增。
- 为什么 ZSet 负责排序，`INCR` 负责严格序号。

## 7. 与下一阶段的衔接

阶段一解决了“数据应该怎么存”。

阶段二要解决“并发下怎么保证同一件事不会被多个实例同时做”。

这会直接对应 RTC 项目里的当前实现：

```text
RTC:DELIVERY:RETRY:LOCK
```

当前项目用 `setIfAbsent` 手写重试锁。阶段二会重点讲：

- `SET NX` 锁到底解决了什么问题。
- 锁过期时间设置短了会发生什么。
- 业务执行超过锁 TTL 为什么危险。
- Redisson 的 `RLock` 和看门狗机制为什么更适合生产场景。
